# 方案 A 设计说明

**目标：** 以最低风险的顺序优化当前减脂应用，先修工程底座和明显缺陷，再逐步增强体验与架构。

## 现状判断

- 当前应用已经具备建档、饮食记录、运动记录、体重记录和仪表盘展示的基础闭环。
- 主要问题集中在三类：工程底座不完整、无障碍和稳定性不足、前端结构不利于继续扩展。
- 因为这是一个单页本地应用，最应该优先保证的是“能稳定运行、能长期维护、每天愿意打开使用”。

## 方案对比

### 方案 A（已确认）

按顺序推进：

1. 工程底座
2. 明显缺陷
3. 体验优化
4. 架构优化

优点：

- 见效快
- 回归风险低
- 每一步都可独立验证

缺点：

- 不会一次性把架构重做干净

### 方案 B

先做整体重构，再补体验。

优点：

- 结构会更整齐

缺点：

- 返工风险高
- 不适合当前体量

## 已确认的执行顺序

### 第一阶段：工程底座 + 明显缺陷

- 补齐缺失的 eslint 依赖和可执行校验
- 清理 Vite 模板残留和项目元信息
- 修复可访问性问题
- 修复不稳定和明显逻辑问题

### 第二阶段：体验优化

- 增加行动提示
- 优化录入效率
- 增强连续使用感

### 第三阶段：架构优化

- 拆分状态
- 给记录项增加稳定 id
- 懒加载重模块
- 收敛样式体系

## 验收标准

- `npm run lint` 可以正常运行
- `npm run build` 可以通过
- 页面标题、语言、说明文档不再是模板默认值
- 主要可点击元素具备基本无障碍语义
- 仪表盘和记录流程文案更稳定、更可信
